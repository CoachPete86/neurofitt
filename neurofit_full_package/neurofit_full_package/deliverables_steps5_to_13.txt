Step 6 — Analytics + Trust
==========================

This deliverable outlines how to implement analytics ethically, build trust with users and meet accessibility and performance standards.

### 1 — GA4 embed snippet and event tracking

Place the following code in the `<head>` of each page, replacing `G-XXXXXXXXXX` with your Google Analytics 4 measurement ID. The snippet is based on Google’s recommended implementation【358122354923729†L117-L126】 and includes consent‑initialised settings:

```html
<!-- Google Analytics 4 (gtag.js) -->
<script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXXXXXXX"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}  
  // Set default consent to denied until user makes a choice
  gtag('consent', 'default', {
    ad_storage: 'denied',
    analytics_storage: 'denied'
  });
  gtag('js', new Date());
  gtag('config', 'G-XXXXXXXXXX', { 'anonymize_ip': true });
  // Example of custom event tracking: record button clicks
  function trackButton(label) {
    gtag('event', 'button_click', { 'event_category': 'engagement', 'event_label': label });
  }
</script>
```

Call `trackButton('buy_starter')` or similar when users click CTA buttons to monitor funnel performance. Always anonymise IP addresses and avoid sending personally identifiable information.

### 2 — Cookie consent banner text

The cookie banner informs users about optional analytics cookies. It should:

* State that essential cookies are used to make the service work; analytics cookies are only set with consent.
* Offer clear **Accept** and **Reject** buttons; once selected, hide the banner and store the choice for a year.
* Not block access to content or be fixed over the page; use non‑sticky positioning as recommended by GOV.UK【727433046591268†L409-L434】.
* Provide a link to the Cookie Policy for more details.

Example banner text (included in the site):

> “We use essential cookies to make our service work and, with your consent, analytics cookies to understand how you use our site. You can accept or reject analytics cookies.”

### 3 — Accessibility checklist (WCAG 2.2 AA)

Use this checklist during development and QA to meet Web Content Accessibility Guidelines (WCAG) 2.2 AA:

1. **Text alternatives:** Provide descriptive `alt` text for images; use `<figcaption>` or visible captions for complex media.
2. **Structural headings:** Use semantic headings (`<h1>…<h6>`) in logical order; include a skip‑navigation link at the top of each page.
3. **Colour contrast:** Ensure text and interface components meet minimum contrast ratios (4.5:1 for normal text, 3:1 for large text). Avoid relying solely on colour to convey information.
4. **Keyboard navigation:** All interactive elements (links, buttons, form controls) must be operable via keyboard alone. Provide visible focus states.
5. **Forms and labels:** Associate labels with form controls using `for` and `id`; describe what each input expects; indicate required fields.
6. **ARIA landmarks and roles:** Use `<nav>`, `<main>` and `<footer>` to support screen readers; supplement with ARIA roles where necessary.
7. **Animations and flashing:** Avoid animations that flash more than three times per second. Provide controls to pause or disable animations.
8. **Error prevention:** Validate input and provide clear, accessible error messages. Avoid irreversible actions without confirmation.
9. **Responsive design:** Ensure layout reflows correctly on small screens; avoid horizontal scrolling. Test using screen‑reader and zoom tools.
10. **Page language:** Declare `lang="en"` on `<html>` and update if content changes language (e.g. in the Superset Builder when practising a foreign language).

### 4 — Performance checklist

1. **Optimise images:** Use SVG for logos and icons; compress JPEG/PNG; specify width and height to prevent layout shifts; use lazy loading (`loading="lazy"`).
2. **Minify CSS and JavaScript:** Combine and minify files; remove unused rules; use `defer` or `async` for scripts where appropriate.
3. **Preload critical assets:** Preload fonts (`<link rel="preload" as="font" href="/assets/fonts/…">`) and above‑the‑fold images to avoid delays.
4. **Use caching headers:** When hosting, enable caching for static assets (e.g. one year). In Netlify this can be configured via `_headers` file.
5. **Reduce third‑party scripts:** Only include necessary analytics and payment scripts; load them after consent. Monitor performance impact.
6. **Validate HTML and CSS:** Use the W3C validator to ensure markup is well‑formed; fix any errors that could hinder browsers’ rendering optimisations.

Step 13 — Deployment & Launch Instructions
========================================

### 1 — Hosting on GitHub Pages

1. Create a new repository (e.g. `neurofit-site`) on GitHub under your account.
2. Push the contents of the `neurofit_site` folder to the repository’s main branch (or a `gh-pages` branch if you prefer). Ensure the `.html`, `.css`, `.js`, `.svg` and image files are included.
3. In the repository settings, enable GitHub Pages by selecting the branch containing the site (usually `main`) and the `/root` folder. GitHub will publish the site at `https://username.github.io/neurofit-site/`.
4. Test navigation, forms and cookie banner. Note that Stripe and GA4 will not work without live keys.

### 2 — Hosting on Netlify

1. Sign in to Netlify and click **New site from Git**.
2. Connect your GitHub repository and select `main` as the branch. Set **Publish directory** to `neurofit_site` (if the folder is at root) or simply leave blank if you copy the contents directly.
3. Netlify will build and deploy the static site automatically. It provides a free HTTPS‑enabled domain (e.g. `neurofit.netlify.app`).
4. To use a custom domain, go to **Domain management** and add your domain. Update your DNS records to point to Netlify’s servers (CNAME or A records). Netlify will generate an SSL certificate automatically.

### 3 — Adding Stripe keys and GA4 IDs

1. Log in to your Stripe dashboard and create a **publishable key** and **secret key** for your production environment.
2. In `/assets/js/stripe.js`, replace the placeholder publishable key and price IDs with your actual values. Ensure each button has the correct price ID for the corresponding tier.
3. Replace `G-XXXXXXXXXX` in `ga4.js` and the GA snippet with your real measurement ID. Verify that events are firing by checking the Realtime view in the GA dashboard.
4. For Netlify deployments, store secret keys as environment variables in the build settings and reference them using Netlify Functions if you need server‑side calls. Never expose secret keys in client‑side code.

### 4 — DNS + SSL setup

* **GitHub Pages:** For a custom domain, add a `CNAME` file containing your domain (e.g. `www.neurofit.co.uk`) in the repository’s root. Then create a CNAME record at your registrar pointing to `username.github.io`.
* **Netlify:** After adding your domain, Netlify provides DNS instructions. Update your registrar’s nameservers or add A/CNAME records as directed. Netlify issues SSL certificates automatically via Let’s Encrypt.

### 5 — Cookie consent check

After deploying, open your site in a private browser window:

1. Confirm that the cookie banner appears on the first visit and does not scroll with the page (no sticky positioning).
2. Use browser dev tools to verify that no GA cookies (e.g. `_ga`) are set before clicking **Accept**. After acceptance, `_ga` should appear and events should send.
3. Ensure preferences persist across pages and visits for at least one year. Rejecting cookies should block GA scripts entirely.

### 6 — Publishing the workbook as a Google Sheet template

1. Create a master version of the workbook in your Google Drive.
2. Share it using the **Anyone with the link can view** permission. Copy the share link.

4. Alternatively, publish the workbook to the web (File → Publish to the web) to embed read‑only versions of specific sheets in the guide.
